## Монолит vs микросервисы

В данном проекте я решил использовать монолитную архитектуру, потому что это оптимальный выбор для старта и небольших проектов.

Вот основные плюсы этого подхода:

- Скорость разработки. Вся логика в одном месте проще писать, отлаживать и деплоить. Не нужно настраивать межсервисное взаимодействие, очереди и контракты API между компонентами
- Простота развёртывания. Один репозиторий, один Dockerfile. Нет необходимости в оркестрации (Kubernetes)
- Производительность без накладных расходов. Вызовы между модулями происходят внутри одного процесса

Микросервисы подходят для больших проектов, где необходимо физическое разделение на модули. 
Как переходный вариант я часто использую модульный монолит. 
Т.е в рамках одного приложения я делаю такую структуру, которая логически похожа на микросервисы, что бы в последствии при необходимости было проще разнести проект на реальные изолированные микросервисы.


## Концепт брокеров сообщений

Брокеры сообщений идеально подходят для коммуникации между микросервисами. Это асинхронная шина, которая позволяет сервисам общаться, не вызывая друг друга напрямую.
Сервис-издатель публикует событие в очередь и сразу продолжает работу в то время как сервис-подписчик забирает сообщение, когда готов. Издатель не знает о получателях и не ждет ответа. Сервисы не знают адреса друг друга и сообщения сохраняются при падении потребителя. Очередь работает как буфер при пиках нагрузки.

В монолите тоже можно применять брокеры. Напрмер, в нашем случае мы могли бы сделать асинхронное логирование:

- Вместо синхронной записи в БД сервис шлёт сообщение в очередь
- Отдельный сервис накапливает батч и пишет его одним INSERT
- При падении БД сообщения копятся в очереди и не теряются

## Логирование и мониторинг

В production-системе сбор логов мог бы быть организован следующим образом:

Логи запросов и ответов API пишутся в JSON-файлы например в директорию logs/api/ с ротацией по дням (например, api-2026-05-26.log).
Для чтения и визуализации логов для нашего небольшого проекта я бы использовал стек: Promtail, Loki, Grafana.

- Promtail читает JSON-файлы логов, парсит их и отправляет напрямую в Loki
- Loki индексирует метаданные логов
- Grafana визуализирует логи из Loki

Для нашего учебного задания я сделал простой logger сервис, который принимает POST запросы через виртуальную сеть в docker и выводит в Logs docker контейнера моковый INSERT запрос к БД.


## Обзор инструментов MLops 

DVC (Data Version Control) отвечает за воспроизводимость данных и пайплайнов. Предназначен для версионирования больших файлов (датасеты, веса моделей) через привязку к Git, отслеживание зависимостей между этапами обработки данных и обучением. Это гарантирует, что конкретный коммит в Git всегда точно воспроизведет ту же самую модель с теми же исходными данными.

MLflow (Tracking) отвечает за организацию процесса экспериментирования. Предназначен для логирования параметров, метрик и артефактов в ходе каждого запуска. Это позволяет сравнивать попытки между собой, находить лучшие комбинации гиперпараметров и вести историю «что и как мы обучали» в удобном веб-интерфейсе.

## Бизнес-метрики

С точки зрения бизнеса, модель нужно оценивать не просто по точности предсказаний, а по её влиянию на операционные и финансовые показатели.

Вот пример такой бизнес метрики:

Доля автоматически одобренных заявок. 

Эта метрика показывает, насколько модель снижает операционную нагрузку и ускоряет процесс, исключая человека из цепочки принятия решений. Мы считаем процент заявок, решение по которым принимается полностью автоматически без участия сотрудника. Рост этой метрики означает, что мы сокращаем расходы на ФОТ экспертов и время выдачи решения.

Расчет: 
STP = (N_auto_approve + N_auto_reject) / N_total

где:
N_auto_approve - одобренные моделью заявки 
N_auto_reject - отклоненные моделью заявки 
N_total - общее количество всех поступивших заявок
